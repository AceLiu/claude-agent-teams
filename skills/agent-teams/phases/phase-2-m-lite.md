# Phase 2 M-Lite — M 级简化设计（同时替代 Phase 1+2）

> **本路径同时替代 Phase 1（需求分析）和 Phase 2（设计）**，适用于需求足够清晰的 M 级项目。
> 需求清晰时（G-xxx 可直接提取、无歧义、技术栈已知），Architect 一次性产出 spec-lite.md。
> M-lite 中 TC 骨架由 Architect 在 spec-lite.md 内一并产出（任务量小，不值得单独派 Tester）。标准 M/L 级的 TC 由 Tester 在 Phase 3 编写。

## 适用条件

- ✅ 用户给了明确功能点 + 技术栈已知 + 无业务歧义
- ❌ 需求模糊、涉及多方协调、有业务术语需确认 → 走标准 `phase-2-design.md`

## 文档深度（v4.1.0 新增）

M-lite 章节按**需求复杂度**裁剪，不再全量必填：

| 需求特征 | 文档深度 | 必填章节 | 可省略 |
|---------|---------|---------|--------|
| **简单变更**：加字段/改逻辑/加校验，AC ≤ 5，涉及 ≤1 个接口 | **精简** | 需求描述 + 验收标准 + 接口契约 | UI 契约(无 UI 时)、TC 骨架(AC→TC 直接映射)、风险、技术方案、追溯矩阵 |
| **标准变更**：新功能/新页面，AC 6-10，涉及多接口 | **标准** | 全部章节 | solution-design.html |
| **复杂变更**：多模块联动、有技术风险 | **退出 M-lite** | → 走标准 Phase 1 + Phase 2 | — |

### 精简深度规则

当 Leader 判定需求为"简单变更"时，在 Architect prompt 中附加：

```
文档深度：精简模式。
- 只填写：需求描述、验收标准、接口契约
- 省略：风险与缓解（无已知风险时）、技术方案（方案显而易见时）、追溯矩阵（AC ≤5 时）
- TC 骨架：每条 AC 直接映射 1 条 TC，不需要完整 10 字段，只填 ID/关联AC/标题/预期
- UI 契约：无 UI 变更时省略
```

**判定标准**：满足以下全部 → 精简深度：
1. AC ≤ 5 条
2. 涉及 ≤ 1 个新接口（已有接口的修改不计）
3. 无数据库 schema 变更
4. 无已知技术风险

> **兜底**：Architect 执行中发现复杂度超预期，可自行升级到标准深度，在 spec-lite.md 中补充章节。

## 步骤

1. 加载知识库（同标准版 2.1）
2. 模板匹配（同标准版 1.3）— 匹配到模板时注入 Architect prompt
3. 代码探索（不可跳过）
4. 派发 Architect（合并 prompt，含 PM 需求分析职责）→ 产出 `.team/spec-lite.md`
5. M-lite 验证: `bash validate-spec.sh {project_root} M` + `bash validate-testcases.sh {project_root} M`（无需单独验证 contracts）
6. Leader 审核 spec-lite.md → 用户确认 → Phase 3

### 退出条件

Architect 执行中如果发现以下情况，应中止 M-lite 并回退到标准 Phase 1：
- 需求存在歧义，需要 PM 深入分析
- 涉及多方业务术语需确认
- 技术方案需要 ≥2 方案对比（单方案不够）

Architect 返回 `NEEDS_CONTEXT` 且原因涉及需求歧义时，Leader 应切换到标准 Phase 1 派发 PM。

## spec-lite.md 模板

> 标注 `[精简可省略]` 的章节在精简深度下可整节删除。

```markdown
# Spec Lite: {需求名称}

## 需求描述
（一段话概述）

## 验收标准
| 编号 | 关联需求 | Given | When | Then |
|------|---------|-------|------|------|
| AC-001 | G-001 | ... | ... | ... |

## 接口契约（简版）
| 接口 | Method | 路径 | 主要参数 | 响应 |
|------|--------|------|---------|------|
| API-001 | POST | /api/... | ... | ... |

## UI 契约 [精简可省略：无 UI 变更时]
| 页面 | 路径 | 关键 data-testid |
|------|------|-----------------|
| ... | /... | submit-btn, search-input, ... |

## 测试用例骨架 [精简可简化]
<!-- 精简深度：每条 AC 映射 1 条 TC，只填 4 字段 -->
| ID | 关联AC | 标题 | 预期 |
|----|--------|------|------|
| TC-001 | AC-001 | ... | ... |

<!-- 标准深度：完整 10 字段 -->
| ID | 关联AC | 关联API | 类型 | 级别 | 标题 | 前置 | 步骤 | 预期 | 反向 |
|----|--------|--------|------|------|------|------|------|------|------|
| TC-xxx-0001 | AC-001 | API-001 | 功能 | P1 | ... | ... | ... | ... | 否 |

## 风险与缓解 [精简可省略：无已知风险时]
| ID | 风险 | 缓解措施 |
|----|------|---------|

## 技术方案 [精简可省略：方案显而易见时]
（技术选型、关键模块、数据流。不强制方案对比，可单方案+理由）

## 追溯矩阵 [精简可省略：AC ≤5 时]
| 需求 | 接口 | 验收 | 测试 | 风险 |
|------|------|------|------|------|
```

## 与标准版的差异

| 维度 | 标准版 | M-Lite |
|------|--------|--------|
| 文件数 | spec.md + design.md + contracts.md + testcases.md | spec-lite.md 一个文件 |
| 方案对比 | ≥2 方案 + 5 维评分 | 单方案 + 理由 |
| TC 覆盖率 | ≥70% | ≥70% |
| solution-design.html | 可选 | 不生成 |
| CLAUDE.md | 生成 | 简版（流水线状态 + 关键决策摘要） |
